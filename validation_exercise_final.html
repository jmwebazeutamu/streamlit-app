Input Validation

1
Exercise Overview & Instructions

About This Exercise

In this exercise you will design and apply input validation rules to a web registration form. The goal is to understand how programs prevent incorrect, unsafe, or unrealistic data from entering a system.

You will analyse each field in the form, determine the appropriate validation rule(s), and describe how the system responds to invalid input.

What You Will Do

  1. Identify which fields need validation and the rule type
  2. Choose the correct validation rule type(s) for each field
  3. Write each validation rule in pseudocode
  4. Justify why you chose that rule

Black-Box Validation Functions

For fields requiring pattern matching, assume the system provides these ready-made functions. You only call them — you do not implement them. Each returns True (valid) or False (invalid).

isValidEmail(email) isValidName(name) isValidPhone(phone) isValidUsername(username) isValidPassword(password) isValidReferral(code) isValidDigits(value, n) lookupValue(word, list)
lookupValue(word, allowedList)
→ True if word exists in allowedList
→ False otherwise

Lookup Function Example

Set allowedCountries = ["United Kingdom", "United States", "Canada", "Germany"]

If Not lookupValue(country, allowedCountries) Then
    Display "Country must be selected from the approved list"
    Set isValid = False
End If

Case Study — UniConnect Student Portal

UniConnect is a university online student portal used by a network of UK higher-education institutions. When a new student is admitted, they must register an account on the portal before they can access timetables, submit coursework, or pay fees.

The registration form collects personal details, contact information, security credentials, and academic information. Because the portal handles sensitive personal data and links directly to institutional records, every field must be rigorously validated before data is accepted into the database.

Poor validation has caused problems in the past: students entered incorrect dates of birth that clashed with financial-aid age requirements; duplicate usernames caused login failures; and malformed student IDs broke the internal lookup system. Your task is to design the validation rules that prevent these issues from recurring.

System Requirements

  • All required fields must be completed before submission
  • Student must be at least 16 years old and no older than 80
  • Username: 6–20 characters, letters/digits/underscores only
  • Password: minimum 8 characters, must contain at least one digit
  • Confirm Password must match the Password field exactly
  • Email must follow a valid email format
  • Phone number: 10–15 digits, may include country prefix
  • Date of birth cannot be in the future
  • Enrollment year must be current year or up to 2 years ahead
  • Student ID uses a check digit for error detection
  • Country and Course must be selected from approved lists
  • First/Last name: letters, hyphens, apostrophes only
  • Age entered must be logically consistent with date of birth

Validation Type Reference
Presence
Checks the field is not empty or blank before accepting the data.
Type Check
Verifies data is the correct data type (e.g. integer, date, string).
Range Check
Ensures a value falls within an acceptable minimum and maximum.
Length Check
Confirms the number of characters is within the allowed limits.
Format Check
Validates that data matches a specific pattern (e.g. email, phone).
Lookup (List)
Checks the value exists within a pre-approved set or list.
Cross-Field
Validates a field by comparing it against another field's value.
Check Digit
Uses a calculated digit to detect transcription errors in a code.
Allowed Chars
Restricts input to a defined set of permitted characters only.
Logical / Semantic
Checks that data makes real-world sense (e.g. age matches DOB).
